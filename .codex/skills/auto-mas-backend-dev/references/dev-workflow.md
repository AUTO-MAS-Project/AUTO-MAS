# 开发流程

## 修改前

- 先在 `app/api`、`app/core`、`app/services`、`app/task`、`app/utils` 中搜索相关功能，确认真正归属。
- 修改接口前，先把对应 handler 和 schema 一起读完，不要只改其中一侧。
- 观察周边实现是否已有同类模式，优先跟随仓库现有写法。

## 修改中

- 改动要成组完成。若涉及传输层、schema、运行时逻辑，尽量一轮补齐。
- 除非需求明确改变协议，否则保持现有响应包结构和 WebSocket 消息结构。
- 涉及 `main.py` 和 `app/core/config.py` 的启动、关闭、副作用逻辑时要特别谨慎。
- 跨脚本类型复用逻辑时，优先抽窄小的辅助函数，不要复制粘贴到多个 manager。

## 验证清单

### 必做

- 运行 `python -m compileall main.py app`。

### 改了 API 或 schema 时

- 检查目标路由模块与 `app/models/schema.py` 是否仍可正常导入。
- 确认 handler 返回内容与模型字段仍然一致。

### 改了运行时或配置时

- 回查所有读取 `Config` 的调用点。
- 检查是否还需要同步修改 WebSocket 广播、文件写入、迁移逻辑。
- 如果行为依赖外部工具、Windows API 或本地环境，要明确说明哪些部分无法在当前环境完成验证。

## 完成标准

- 改动结果要让下一个接手的 agent 一眼能看出改了什么、为什么这么改。
- 回复里说明已做的验证和当前环境限制。
- 如果引入了新的后端模式，应放进最合适的公共位置，而不是临时散落在多个文件里。
