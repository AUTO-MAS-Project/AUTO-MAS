# 脚本适配检查清单

## 新增脚本类型时

1. 是否在 `app/models/config.py` 补了对应配置模型。
2. 是否在 `app/models/schema.py` 补了 API 暴露所需模型。
3. 是否在 `app/core/task_manager.py` 增加了类型分派。
4. 是否新增了对应 `manager.py`。
5. 是否补齐 `AutoProxy`、`ManualReview`、`ScriptConfig` 中实际需要支持的模式。
6. 是否补齐 `tools/` 中的脚本类型专属工具。

## 修改 manager 时

1. `check()` 是否仍能提前发现路径、文件、配置缺失问题。
2. `prepare()` 是否仍会锁定配置并备份原始文件。
3. `main_task()` 是否仍按预期遍历用户并 `spawn()` 子任务。
4. `final_task()` 是否仍能解锁、回写、通知、回滚、清理。
5. `on_crash()` 是否仍能把错误传到日志和 WebSocket。

## 修改 `AutoProxy` 时

1. 是否保留用户级校验、日切初始化和运行次数限制逻辑。
2. 是否保留模拟器/客户端启动、配置注入、脚本启动和日志监控的顺序。
3. 是否仍会正确更新用户状态、日志历史记录、`ProxyTimes`、`LastProxyDate` 等数据。
4. 是否保留失败重试、前置错误处理或脚本异常退出后的清理流程。
5. 是否同步检查通知汇总和 `final_task()` 汇总字段是否仍成立。

## 修改 `ManualReview` 时

1. 是否仍会订阅 `Broadcast` 或保持现有交互输入通道。
2. 是否保留问题消息的 `message_id`、问题类型、选项结构和等待逻辑。
3. 是否处理了取消、重试、否决和超时等非理想路径。
4. 是否仍会把人工确认结果写回用户状态、任务状态或相关数据字段。
5. 是否保留窗口显示、模拟器关闭和异常清理流程。

## 修改 `ScriptConfig` 时

1. 是否仍会在写配置前停止旧进程。
2. 是否仍会从 `data/<script_id>/<user_id>/ConfigFile` 读取已有用户配置。
3. 是否保证脚本进程启动后不会自动运行任务。
4. 是否仍会在结束后把配置目录回存到用户配置缓存目录。
5. 是否避免只改脚本目录而漏掉项目内缓存目录，或反过来只改缓存目录而未写回脚本实际运行目录。

## 修改配置相关逻辑时

1. 是否兼容已有 `Default/ConfigFile`、`Temp` 等目录约定。
2. 是否保持不同脚本类型各自的配置文件形态。
3. 是否避免把单一脚本类型特有逻辑误抽成公共逻辑。

## 修改结果汇总或通知时

1. 是否保留成功、异常、等待三类用户统计。
2. 是否保留开始时间、结束时间、脚本名、结果字段。
3. 是否仍能在失败时通过 `Config.send_websocket_message()` 反馈。

## 结束前

- 运行 `python -m compileall main.py app`。
- 检查目标脚本类型相关模块能否正常导入。
- 明确写出已验证部分与未验证部分。
