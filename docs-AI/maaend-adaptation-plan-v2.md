# MaaEnd 适配新计划（复用 MAA 思路）

## 核心策略
- 复用 MAA 的“外部配置优先”模式：启动外部配置流程 -> 回读配置 -> 调度执行。
- AUTO-MAS 不重建 MaaEnd 全量任务/选项模型，避免高维护成本。
- 保持最小改动：优先复用现有配置管理、状态广播、历史与通知链路。

## 新的里程碑顺序
1. I01：配置模型接入（已完成）
2. I02：Schema/API/OpenAPI 接入（已完成）
3. I03：外部配置接入能力（替代“元数据全量解析”）（已完成）
4. I04+I05：运行期参数注入 + TaskManager 分发到 MaaEndManager（已完成）
5. I06：check/prepare/final 生命周期（已完成）
6. I07：执行、日志判定、失败归因、重试（已完成）
7. I08：历史/通知/WebSocket（已完成）
8. I09-I10：前端入口与脚本级最小编辑页（已完成）
9. I11：用户级最小配置页（进行中）
10. I12-I13：联调回归与发布文档

## I03（重定义）
### 目标
- 对齐 MAA：通过外部配置器维护 MaaEnd 配置，AUTO-MAS 只管理必要字段并回读。

### 范围
- 提供启动 MaaEnd 配置流程的后端能力。
- 回读并持久化最小关键字段（控制器、资源、预设、必要运行参数）。
- 保证配置回读幂等，不引入任务/选项硬编码。

### 非目标
- 不做 `interface/tasks/preset` 全量元数据建模。
- 不做复杂动态任务表单生成（放到后续可选优化）。

## 接口与实现约束
- API 风格对齐现有 `scripts` 路由（统一 `OutBase` 错误格式）。
- `Config` 层承载核心逻辑，路由层仅转发与错误包装。
- 运行链路仍在 I05+ 实现；I03 不承诺可执行，仅承诺“可配置并可回读”。

## 验收标准
- 能从 AUTO-MAS 触发一次 MaaEnd 配置流程并成功保存。
- AUTO-MAS 能回读并持久化关键字段，重启后可复现。
- 不破坏 MAA/General 现有行为。

## 风险与对策
- 外部配置格式变动：采用“最小字段回读”策略，降低耦合。
- 配置入口失败：提供明确错误信息与回退提示。
- 与后续执行链路脱节：I04 明确只做最小参数注入与桥接。
- MaaEnd 日志路径差异：默认监控路径统一为 `Info.Path/debug/go-service.log`。
